# Матеріали Для Тексту Курсової: Docker Compose Схема

## 1. Суть реалізованого підходу

У проєкті використано багатофайлову конфігурацію Docker Compose за схемою:

- базовий файл: `docker-compose.yml`
- файл-надбудова для розробки: `docker-compose.dev.yml`
- файл-надбудова для production-подібного запуску: `docker-compose.prod.yml`

Підхід `base + overlays` означає, що спільна інфраструктурна логіка винесена в один базовий файл, а особливості окремих режимів запуску описані в додаткових файлах-надбудовах.

---

## 2. Основна ідея архітектури

Базовий compose-файл містить лише спільні для всіх режимів компоненти:

- опис сервісів `db`, `backend`, `frontend`;
- внутрішню взаємодію контейнерів;
- healthcheck;
- Docker Secrets;
- залежності між сервісами;
- спільні змінні середовища;
- volumes для збереження даних.

Додаткові compose-файли не дублюють увесь стек, а лише змінюють ті параметри, які мають відрізнятися між середовищами.

---

## 3. Роль кожного compose-файлу

### `docker-compose.yml`

Призначення:

- бути єдиною базою конфігурації;
- містити спільний опис сервісів;
- зменшувати дублювання між режимами `development` і `production`;
- задавати стабільну структуру контейнеризованого стеку.

Що включає:

- `db` на базі `postgres:17-alpine`;
- `backend` на базі Node.js + Express + Prisma;
- `frontend` на базі Next.js;
- Docker Secrets для пароля PostgreSQL, JWT secret та OpenAI API key;
- volumes для PostgreSQL та кешів/залежностей;
- healthcheck для PostgreSQL, backend і frontend;
- залежності старту через `depends_on`.

### `docker-compose.dev.yml`

Призначення:

- запуск локального середовища розробки;
- забезпечення швидкої ітерації без повного production-білду після кожної зміни;
- робота з автоматичним перезавантаженням під час змін у коді.

Що змінює відносно базового файлу:

- підключає `Dockerfile.dev` для бекенда і фронтенда;
- додає bind mounts з локальним кодом;
- запускає `nodemon` для бекенда;
- запускає `next dev` для фронтенда;
- публікує зовнішні порти для доступу з локального середовища;
- використовує окремі Docker volumes для `node_modules` і `.next`.

### `docker-compose.prod.yml`

Призначення:

- запуск production-подібного середовища;
- робота з уже зібраними production-образами;
- мінімізація залежності від локальної файлової системи.

Що змінює відносно базового файлу:

- залишає стандартні production `Dockerfile`;
- встановлює `NODE_ENV=production`;
- публікує лише необхідні зовнішні порти застосунку;
- не експонує PostgreSQL назовні;
- не монтує локальний код;
- не вмикає dev-інструменти на кшталт `nodemon` або `next dev`.

---

## 4. Логіка роботи контейнеризованого стеку

У будь-якому режимі стек складається з трьох сервісів:

- `db`
- `backend`
- `frontend`

Порядок запуску:

1. Стартує контейнер PostgreSQL.
2. Виконується перевірка готовності БД через `pg_isready`.
3. Після того стартує backend.
4. Backend під час запуску виконує Prisma migration deploy.
5. Після успішного healthcheck backend стартує frontend.

Така схема запобігає ситуації, коли:

- backend стартує раніше, ніж готова база даних;
- frontend починає працювати раніше, ніж готовий API.

---

## 5. Чому обрано саме схему `base + overlays`

### 5.1. Зменшення дублювання

Якби для `dev` і `prod` були створені два повністю окремі compose-файли, довелося б дублювати:

- опис сервісів;
- secrets;
- healthcheck;
- volumes;
- мережеву логіку;
- більшість змінних середовища.

Схема `base + overlays` дає змогу уникнути дублювання та зменшує ризик розсинхронізації конфігурацій.

### 5.2. Простота супроводу

При зміні спільної логіки достатньо оновити лише базовий файл, а не декілька незалежних конфігурацій.

Це спрощує:

- технічну підтримку;
- розширення інфраструктури;
- виправлення конфігураційних помилок;
- оновлення секретів, healthcheck і залежностей.

### 5.3. Логічне розділення режимів

Середовища `development` і `production` мають різні вимоги:

#### Для `development`

Потрібно:

- швидко бачити зміни коду;
- не витрачати час на повний production build;
- мати зручний цикл редагування і перезапуску.

#### Для `production-подібного` режиму

Потрібно:

- використовувати стабільні зібрані образи;
- не залежати від bind mounts;
- наблизити поведінку стеку до реального сценарію розгортання.

Саме тому файли-надбудови дають змогу чітко відділити ці два сценарії.

### 5.4. Відповідність практикам контейнеризації

Багатофайлова конфігурація Compose є типовим підходом у реальних проєктах, де:

- є кілька режимів запуску;
- існує спільне конфігураційне ядро;
- потрібно масштабовано підтримувати інфраструктуру.

Отже, обраний підхід є не лише технічно зручним, а й методологічно обґрунтованим.

---

## 6. Пояснення dev-режиму

Dev-режим орієнтований на локальну розробку.

Його характерні риси:

- вихідний код монтується з хоста в контейнер;
- зміни відображаються без повного перевидання production-образу;
- backend працює через `nodemon`;
- frontend працює через `next dev`;
- `.next` і `node_modules` винесені у Docker volumes, що зменшує проблеми з платформозалежними залежностями.

Практичне призначення:

- розробка функціоналу;
- налагодження;
- швидке внесення змін;
- тестування UI та API у процесі розробки.

---

## 7. Пояснення production-подібного режиму

Production-подібний режим орієнтований не на розробку, а на стабільний запуск.

Його характерні риси:

- код уже включено до образу;
- немає bind mounts;
- фронтенд запускається як production Next.js server;
- бекенд запускається як production Node.js service;
- конфігурація ближча до реального деплой-сценарію.

Окремо важливо враховувати поведінку змінних фронтенда:

- `NEXT_PUBLIC_API_URL` і `NEXT_PUBLIC_SOCKET_URL` у production передаються як аргументи збірки під час `docker build`;
- ці значення вбудовуються в клієнтський застосунок і не змінюються простим перезапуском контейнера;
- для серверних запитів усередині контейнера фронтенда використовується змінна середовища `INTERNAL_API_URL=http://backend:5000`.

Практичне призначення:

- демонстрація завершеного застосунку;
- перевірка поведінки production-збірки;
- тестове розгортання;
- підготовка до подальшого хостингу.

---

## 8. Використання Docker Secrets

У конфігурації застосовано Docker Secrets для чутливих даних:

- пароль PostgreSQL;
- `JWT_SECRET`;
- `OPENAI_API_KEY`.

Причини використання:

- секрети не зберігаються безпосередньо в compose-файлах;
- чутливі значення відокремлено від загальної конфігурації;
- структура краще відповідає безпечним практикам роботи з конфіденційними даними.

Фактично значення секретів передаються через файли в директорії `secrets/`, а в контейнерах доступні через `/run/secrets/...`.

---

## 9. Healthcheck як елемент надійності

У проєкті реалізовано healthcheck для всіх ключових сервісів:

- PostgreSQL перевіряється через `pg_isready`;
- backend перевіряється через HTTP endpoint `/health`;
- frontend перевіряється через HTTP-відповідь з порту застосунку.

Це дає такі переваги:

- керований порядок запуску;
- коректне визначення готовності сервісу;
- зменшення кількості помилок на старті стеку;
- підвищення стабільності контейнеризованого середовища.

---

## 10. Prisma у контейнерній схемі

Backend інтегровано з Prisma таким чином:

- під час збірки образу виконується `prisma generate`;
- під час старту контейнера виконується `prisma migrate deploy`.

Практичний сенс:

- Prisma Client завжди присутній у контейнері;
- база даних приводиться до актуальної схеми перед запуском API;
- виключається потреба вручну запускати міграції перед стартом стеку.

---

## 11. Практичні переваги розділення dev/prod

Переваги підходу:

- немає дублювання всієї Docker-конфігурації;
- спільна логіка зберігається в одному місці;
- режими розробки і production-подібного запуску розділені чітко;
- зменшується ймовірність конфігураційних помилок;
- підтримка проєкту стає простішою;
- production-образи можна оптимізувати окремо від dev-сценарію;
- підхід відповідає сучасним практикам роботи з Docker Compose.

## 11.1. Ознаки Production-Орієнтованої Контейнеризації

Для сильнішого production-рівня в проєкті також застосовано такі рішення:

- backend збирається через багатоступеневий Dockerfile;
- у backend встановлюються лише production-залежності через `npm ci --omit=dev`;
- Prisma Client генерується під час build-етапу, а міграції застосовуються під час старту контейнера;
- backend і frontend у production запускаються не від `root`, а від користувача `node`;
- frontend використовує Next.js standalone-артефакт для виконання;
- frontend розділяє публічні URL, що вбудовуються під час збірки (`NEXT_PUBLIC_*`), і URL середовища виконання для SSR (`INTERNAL_API_URL`);
- у фінальний образ фронтенда копіюються лише файли, необхідні для запуску production-збірки.

## 12. Готові тези для академічного тексту

### Варіант 1. Короткий абзац

У межах проєкту контейнеризація реалізована за допомогою багатофайлової конфігурації Docker Compose. Базовий файл `docker-compose.yml` містить спільний опис сервісів, мережевої взаємодії, секретів та механізмів перевірки готовності контейнерів. Для адаптації системи до різних сценаріїв експлуатації додатково використано файли-надбудови `docker-compose.dev.yml` та `docker-compose.prod.yml`, які задають параметри відповідно для середовища розробки та production-подібного запуску. Такий підхід зменшує дублювання конфігурації, спрощує супровід системи та забезпечує чітке розділення режимів роботи.

### Варіант 2. Розширений абзац

Для організації контейнеризованого середовища в проєкті застосовано принцип багатофайлової конфігурації Docker Compose, що базується на поєднанні одного базового конфігураційного файлу та двох файлів-надбудов. Базовий файл `docker-compose.yml` містить спільну для всіх режимів структуру системи: сервіси бази даних, серверної та клієнтської частин, механізми healthcheck, Docker Secrets і логіку взаємозалежності контейнерів. Файл `docker-compose.dev.yml` орієнтований на локальну розробку й забезпечує використання bind mounts, `nodemon` та `next dev`, тоді як `docker-compose.prod.yml` призначений для production-подібного запуску на основі попередньо зібраних образів без монтування локального коду. Застосування такого підходу дозволяє уніфікувати спільну конфігурацію, зменшити кількість дубльованих налаштувань та підвищити керованість контейнерної інфраструктури.

### Варіант 3. Ключова теза списком

- використано багатофайлову конфігурацію Docker Compose;
- спільні параметри винесено в базовий compose-файл;
- dev-режим і prod-режим реалізовано через файли-надбудови;
- така схема мінімізує дублювання та спрощує супровід;
- підхід відповідає практикам розділення середовищ розробки й розгортання.

## 13. Фактичні команди запуску

### Dev

```bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up --build
```

### Prod-style

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up --build -d
```

## 14. Що важливо не загубити в підсумковому тексті

У фінальному тексті для курсової бажано обов’язково відобразити:

- що конфігурація розділена на базову і режимні надбудови;
- що dev і prod мають різні цілі;
- що спільна конфігурація не дублюється;
- що використано Docker Secrets;
- що реалізовано healthcheck;
- що Prisma інтегрована у запуск backend-контейнера;
- що такий підхід підвищує підтримуваність та надійність системи.

## 15. Коротка структурна схема

```text
docker-compose.yml
├─ спільні сервіси
├─ secrets
├─ healthcheck
├─ depends_on
└─ volumes

docker-compose.dev.yml
├─ монтування локального коду
├─ nodemon
├─ next dev
└─ dev Dockerfile

docker-compose.prod.yml
├─ production-подібний режим
├─ production Dockerfile
└─ стабільний запуск готового середовища
```
